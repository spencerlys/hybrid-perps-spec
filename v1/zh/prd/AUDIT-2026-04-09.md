# PRD 审计报告（2026-04-09）

> 审计范围：全部 PRD（12 个模块）、场景文档（4 类 23 个场景）、交易流程（4 个流程）、mindmap、CLAUDE.md、README.md
> 参考资料：Notion「下单场景」、Notion「模式本质分析」

---

## 一、核心设计方向确认

**当前 PRD 实现的是方案二：内部对赌 + 阈值对冲（Order-level Internalization）。**

这与设计意图一致：平台默认对赌（B-book），选择性丢过去 HL 对冲。撮合系统（方案三）不在当前阶段。

---

## 二、关键问题清单

### 🔴 P0 — 架构/阶段规划问题

#### 问题 1：对冲引擎不应在 Phase 3

**现状**：L6 风控与敞口管理层（含对冲引擎）整体在 Phase 3。
**问题**：Phase 2 路由引擎上线后就会产生 INTERNAL 仓位，净敞口立即开始累积。如果对冲引擎要等到 Phase 3 才开发，Phase 2 期间平台将完全裸露在方向性风险中。
**建议**：对冲引擎（Hedge Engine）应独立拆出，提前到 Phase 2 开发。至少 Phase 2 灰度期间就需要基础对冲能力。

具体拆分建议：
- **Phase 2 交付**：净敞口实时计算 + 对冲引擎基础版（手动/自动触发对冲到 HL）+ 风险准备金基础管理
- **Phase 3 交付**：路由模式联动自动切换 + 对冲策略优化 + 完整风控仪表盘

#### 问题 2：内部订单簿与对冲系统应明确分层

**现状**：PRD 中 L3（对赌执行）和 L6（风控敞口）的关系模糊。L3 含"限价单内部订单簿"，L6 含对冲引擎，但没有清晰的边界定义。
**问题**：订单簿是记录层（记录哪些 INTERNAL 限价单待成交），对冲系统是风控层（当敞口超阈值时去 HL 开反向仓位）。两者职责完全不同，不应混淆。
**建议**：
- L3 内部订单簿：仅负责 INTERNAL 限价单的待成交队列管理，用 HL 价格触发成交
- L6 对冲引擎：独立模块，当净敞口达阈值时向 HL 下对冲单，与 L3 的限价单队列无关
- 流程明确为：**订单进入 L3 内部记账 → L6 实时聚合净敞口 → 超阈值时 L6 触发对冲 → 对冲指令走 L4 到 HL**

#### 问题 3：mindmap 中 L3 模块名称错误 — "内部撮合执行"

**现状**：mindmap/index.html 中 L3 的第一个模块名为 `内部撮合执行`。
**问题**：方案二是对赌（Internalization / B-book），不是撮合（Matching）。"撮合"是方案三的概念，暗示用户之间互为对手方。当前模型中平台是唯一对手方，没有撮合行为。
**建议**：改为 `对赌执行引擎` 或 `内部对赌执行`。

### 🟡 P1 — 场景覆盖遗漏

#### 问题 4：巨鲸下单拆单场景缺失

**Notion 场景 5**：用户下 $2,000,000 市价单 → 应拆单（TWAP/VWAP），分批路由 HL，控制滑点。
**PRD 现状**：未覆盖。05-hl-execution.md 提到部分成交处理，但无主动拆单逻辑。
**建议**：在 L4 HL 代理执行层新增「大额订单拆单策略」章节：
- 名义价值 > 拆单阈值（如 $500K）→ 自动拆分
- 支持 TWAP（时间加权）和简单均分两种策略
- 每个子单独立下单，汇总后统一回执给用户

#### 问题 5：跨阈值订单拆分场景缺失

**Notion 场景 8**：用户下 $120K，阈值 $100K → $100K 内部对赌 + $20K 路由 HL。
**PRD 现状**：02-routing.md 的路由逻辑是「每笔订单整体判断」，不支持单笔订单拆分到两个路由。
**评估**：这是方案二的高级能力，Notion 自身也标注"方案2很难做到"。MVP 阶段建议整笔路由（当前设计），但应在 PRD 中明确说明此限制，并在 Phase 4 路由优化中预留此能力。
**建议**：在 02-routing.md 增加「MVP 限制说明」章节，明确：
> MVP 阶段不支持单笔订单拆分路由。名义价值超过阈值的订单整笔路由到 HL。跨阈值订单拆分能力预留到 Phase 4。

#### 问题 6：加仓触发对冲的场景路径不完整

**Notion 场景 6**：用户已有 $5K 多仓，再加仓 $30K → exposure 增加 → 可能触发 hedge。
**PRD 现状**：04-internal-execution.md 说"加仓时每笔独立路由"，但对于加仓导致净敞口突破对冲阈值的场景，没有在 scenarios 中体现完整链路。
**建议**：在 scenarios/01-order-routing.md 增加一个场景：SC-RT-007「连续加仓触发对冲」。

#### 问题 7：用户对敲场景定义需澄清

**Notion 场景 3**：用户 A 买 $50K，用户 B 卖 $50K → 方案二下两者都是平台对赌。
**PRD 现状**：已正确处理（两笔各自与平台对赌，净敞口自然抵消），但没有显式场景说明这种"自然对冲"效果。
**建议**：在 scenarios 或 07-risk-management.md 中增加说明：当多空方向的用户分别对赌时，净敞口自然趋零，这是方案二的核心盈利条件。

## 五、修正后的 Phase 规划建议

| 阶段 | 名称 | 核心交付物 |
|------|------|-----------|
| Phase 1 | 基础接入 | 账户体系 + HL API 全量封装（R0 中继层）+ 市场数据透传 + 多链归集 + 提现保障 |
| Phase 2 | 路由 + 对冲 | 订单路由引擎 + HL 代理执行 + **对冲引擎基础版 + 净敞口监控 + 风险准备金**（灰度：全量转 HL） |
| Phase 3 | 对赌上线 | 对赌执行层 + 统一清算 + 全仓/逐仓保证金 + HL 计算复刻 + 偏差兜底 + 风控仪表盘 |
| Phase 4 | 优化迭代 | 路由阈值调优 + 对冲策略优化 + 大额拆单 + 跨阈值拆分路由（预留） |

**关键变化**：对冲引擎从 Phase 3 提前到 Phase 2，与路由引擎同步交付。理由：有路由就有敞口，有敞口就必须能对冲。
