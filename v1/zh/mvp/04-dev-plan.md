# 多团队开发计划（小时制）

本文档规划化永续期货平台（方案二）的多团队并行开发工时与里程碑。基于3.5周交付周期与6组并行团队架构。

---

## 一、团队划分（6组）

| 组号 | 名称 | 人数建议 | 负责模块 |
|------|------|---------|---------|
| **T1** | 账户基础组 | 2后端+1前端 | L1账户资产, 充提币, 多链归集, 热冷钱包 |
| **T2** | 交易核心组 | 3后端 | L2订单路由, L3对赌执行, 订单生命周期, 成交记账 |
| **T3** | HL对接组 | 2后端 | R0 API中继, L4 HL代理执行, L7市场数据透传, WebSocket |
| **T4** | 风控引擎组 | 2后端 | L5保证金清算, L6敞口监控+对冲引擎, 风险准备金, 熔断 |
| **T5** | 结算对账组 | 1后端+1数据 | L8结算对账, PnL计算, 资金费结算, 三方对账 |
| **T6** | 后台运营组 | 1后端+1前端 | L9后台管理, 仪表盘, RBAC, 审计日志 |

**总人力**: 约14人，其中后端10人，前端2人，数据1人。

## 七、开发落地要点

### 1. 代码组织与分支策略
- 按模块建立独立代码仓库: `repo-l1-account`, `repo-l2-routing`, 等
- 主分支: `main`(生产), `release/phase-X`(阶段分支), `dev`(开发)
- 所有新代码通过PR进行，需2名审核者批准
- 灰度部署: 金丝雀(5%), 小流量(20%), 全量(100%)

### 2. 接口规范与联调
- 所有跨组接口使用OpenAPI 3.0规范，自动生成SDK
- 定义6组间的核心接口(T1→T2: OrderCreateRequest, T2→T3: RoutingDecision等)
- 周一与周五进行跨组接口对齐会议
- 使用Mock Server进行并行开发

### 3. 数据库设计与同步
- T3负责核心DB Schema设计(统一字段类型/索引)
- 关键表: users, wallets, orders, positions, liquidations, hedge_orders
- 使用Liquibase进行迁移管理，所有Schema变更需版本控制
- 每周数据库同步检查

### 4. 测试覆盖与QA节奏
- 单元测试: ≥80%代码覆盖(每个组自己负责)
- 集成测试: 每个Phase周期测试所有跨组链路
- 压力测试: Phase 2达到1000订单/秒, Phase 3达到5000清算/小时
- QA测试清单: 参见《验收清单》06-acceptance.md

### 5. 监控与告警体系
- 每个组配置Prometheus指标(延迟/错误率/吞吐)
- 关键告警(P0): HL断连, HL账户保证金<150%, 清算失效
- 告警响应: P0 5分钟内响应, P1 30分钟内响应
- 建立值班表: 一周一轮, 每轮2人(后端+运维)

### 6. 文档与知识共享
- 每个模块编写API文档 + 部署指南
- 周周五下午进行30分钟技术分享(负责人轮流)
- 维护内部Wiki: 架构图、常见问题、故障排查
- Phase完成后整理总结报告

## 十、附录：工时估算方法

```
总工时 = (开发工时 + 联调工时 + 测试工时) × 依赖系数

开发工时估算: 使用Planning Poker法(T-shirt sizing: XS=4h, S=8h, M=16h, L=32h, XL=64h)
联调工时估算: 开发工时 × 20-30% (跨组联调通常为开发工时1/3-1/5)
测试工时估算:
  - 功能测试: 开发工时 × 25-50%
  - 性能测试: 额外开发工时 × 50% (仅Phase 2-3)
  - 灰度验证: 固定20-40h (全组共享)

并行系数:
  - 无依赖任务: 系数 = 1.0 (完全并行)
  - 弱依赖任务: 系数 = 1.1 (少量同步点)
  - 强依赖任务: 系数 = 1.3-1.5 (待前置完成)

总周期 = max(T1-T6各组总工时) / (人数 × 周工时40h) + 同步/缓冲(+10%)
```
